# Dynamic Debounce with Redis TTL

**Time**: 12:47  
**Source**: Issue #7 - Debounce batching verification

## What We Learned
- Dynamic debounce: each new message refreshes Redis TTL, extending the wait period
- Redis lock (`flush_lock:{conversation_id}`) prevents duplicate worker jobs per conversation
- Integration test can verify debounce behavior by checking worker logs without restarting worker
- DRY_RUN config flag can skip Zendesk API calls during testing (but should not be committed)

## How Things Connect
- `[persistence.enqueue_flush]` → sets Redis lock with TTL → `[worker.flush_buffer]` → waits for TTL to expire → batches messages
- Each message call to `enqueue_flush` refreshes the TTL, creating dynamic debounce
- Worker logs show "remaining_seconds" to verify debounce timing

## Key Discoveries
- Worker waited 10s after LAST message (t+17s for messages at t+0, t+3, t+6)
- All 3 messages combined into single flush: `'Message 1...\nMessage 2...\nMessage 3...'`
- 404 error on Zendesk API is expected with fake conversation IDs - debounce logic still verified

## Apply When
- Testing message batching/debounce behavior
- Verifying worker processes messages correctly
- Debugging timing issues in async worker
