# ARQ Worker Function Registration Requires func() Wrapper

**Time**: 16:54  
**Source**: Issue #3 - ARQ worker not processing jobs

## What We Learned
- ARQ workers require the `func()` wrapper for proper job deserialization
- Using raw function references in `WorkerSettings.functions` causes jobs to be enqueued but never executed
- The worker logs show "Starting worker for 1 functions: flush_buffer" but "0 jobs complete" on shutdown

## How Things Connect
- [enqueue_job] relates to [flush_buffer function] because without `func()` wrapper, ARQ cannot deserialize the job
- Pattern [func(flush_buffer, keep_result=0)] enables [proper job execution] when [jobs are queued from API]

## Key Discoveries
- ARQ job serialization includes function metadata that requires the `func()` wrapper to match
- Even though the function name appears in worker logs, the raw function reference cannot be used for deserialization
- The `keep_result=0` parameter makes results expire immediately (fire-and-forget pattern)

## Apply When
- ARQ worker shows registered functions but never executes jobs
- Jobs appear in Redis but worker reports "0 jobs complete"
- Worker logs show function is registered but jobs are never picked up
